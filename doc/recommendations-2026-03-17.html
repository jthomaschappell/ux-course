Website recommendations

7-Day UX Fluency Plan — Generated 2026-03-17

Recommendations from a scan of the codebase and UX patterns. Use as a prioritized checklist to improve the site.

1. Meta & SEO

  • Add <meta name="description" content="..."> in index.html for search snippets and sharing.
  • Add Open Graph tags (og:title, og:description, og:type) for link previews.
  • Replace default /vite.svg favicon with a course-specific icon (and consider an Apple touch icon).

2. Font consistency

  • High index.html loads DM Sans from Google Fonts; variables.css and STYLE_GUIDE.md reference Inter and Sora. Align: either switch the font link to Inter + Sora, or update CSS variables and the style guide to DM Sans.

3. Accessibility

  • Reference panel toggle: add aria-controls pointing to the expandable content container so assistive tech can link button and panel.
  • Progress bar: add aria-valuenow, aria-valuemin, aria-valuemax (and role="progressbar") on the progress fill so screen readers announce percentage.
  • Reference panel when open: add aria-label to the content wrapper so the region is named (e.g. “Reference and resources”).
  • Ensure all form inputs have a single, programmatic label (e.g. id + htmlFor) where labels don’t wrap the control; current forms are mostly fine but worth auditing for dynamic or repeated fields.

4. Performance

  • Consider lazy-loading day content (e.g. React.lazy + Suspense per day) so only the active day’s form and copy are in the initial bundle; low priority if the app stays small.
  • Persistence: usePersistedState writes to localStorage on every update. For heavy forms (e.g. long textareas), debounce or batch writes (e.g. 300–500 ms after last change) to reduce write frequency.
  • Fonts: preconnect to Google Fonts is already in place; consider font-display: swap in the font URL if not already set.

5. UX & layout

  • Reference panel is fixed at the bottom and can cover the “Review artifacts” / next-action target when open. Consider a max-height (e.g. 50vh is already there), a “collapse to bar only” state, or scroll-into-view that accounts for the panel so the CTA isn’t hidden.
  • Day tabs: when switching days, optionally scroll the main content to the top so the user sees the new day’s intro immediately.
  • Empty state: “No tasks for this day” is good; consider a short empty state for artifact forms (e.g. “Complete the tasks above, then fill your artifacts here”) where it helps.

6. Data & persistence

  • Storage schema is versioned (version: 1); document the migration path for future versions (e.g. in usePersistedState.ts or a small migration module) so adding new fields or renaming keys is safe.
  • Optional: add an “Export progress” / “Import progress” (e.g. JSON download and file upload) so users can back up or move data between devices.

7. Code & maintainability

  • DayContent uses class bento-grid but there is no .bento-grid rule in the scanned CSS; either define it (e.g. grid layout for day sections) or remove the class to avoid confusion.
  • Repeated list keys: ReferencePanel and similar use index-as-key for static lists; prefer stable keys (e.g. resource name or id) if list order might change or items are dynamic.

8. Content & course design

  • Cross-reference doc/course-content-recommendations.md for content-specific improvements (study items, Day 1 clarity, reference panel copy, artifact scaffolding, “Good looks like,” etc.).
  • Ensure reference resources stay current (Godly and others) and that external links are periodically checked.

9. Testing & QA

  • README already includes a manual QA checklist; consider adding a note about testing with a screen reader (e.g. VoiceOver) and keyboard-only for the tabs, reference toggle, and one full artifact form.
  • If the app is deployed, add a smoke test or minimal E2E (e.g. load app, switch day, toggle reference panel, tick one task) to catch regressions; see doc/e2e-tests.md for scenarios.
